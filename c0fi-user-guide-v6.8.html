1. What C0fi is

C0fi is a visual workflow engine in the spirit of n8n, rebuilt around one inversion: the LLM is not a node inside the workflow — it is the engine that builds and runs the workflow. Your local model (via Ollama) sits in the loop four ways:

  • Decide — Decision nodes route branches; Critic nodes judge quality.
  • Orchestrate — the Orchestrator panel sees the whole canvas and can rewire it.
  • Create — describe a workflow in plain language and C0fi constructs the nodes and wires.
  • Build — it edits, extends, and re-runs flows mid-conversation, including flows you drew by hand.

Everything runs in one HTML file against your local Ollama. No accounts, no telemetry. One small local helper (c0fi_server.py) upgrades web search and page reading from real result pages; flows export as JSON files you own, or save to a browser-local library.

New in v6.3 → v6.8 — Auto BrewsFlows can now run on a schedule, by themselves. The new ⏰ Auto Brews button (next to Clear canvas) deploys whole flows to a scheduled list — every N minutes, daily at a time, or weekly on the days you pick (Mon/Wed/Fri at 09:00, every Monday at 1 am, and so on) — and they brew automatically while the tab is open (§11). Deploy as many as you want; each is pausable, deletable, and runs independently. Every run can auto-save its result to a .txt file, and there's a per-brew ⤓ Save .txt for the last result. A single engine lock means brews never collide — a busy engine holds and retries instead. Deleting a brew that's running stops it for you. See §11 for the whole workflow.
New in v6.2A model pool — Engine settings now has four model slots (Model, Verifier, Model C, Model D), so different nodes can run on different models (§2, §5) · a node's model override is now a live dropdown of the models your engine actually has, plus the four slots — no more hand-typing names (§5) · the Verifier bench demo's three judges read the Verifier / Model C / Model D slots and are labelled Judge 1–3, so setting three real models gives you genuinely independent judging (§12) · an exported app that lands with no engine shows a short "run me on Ollama" hint instead of failing silently (§10) · the status line no longer overflows when a message is long — the full text stays in the Brew Log.
New in v6.1A ⇉ Gather join node (§5) that waits for several producers and hands you one array · undo / redo and node copy-paste on the canvas (§4) · engine settings now persist across reloads — no reconnecting every session · Stop cancels an in-flight generation instantly instead of at the next node (§8) · a waiting 💬 Interaction node now shows a "✍ your turn" badge and a reply bar on the canvas itself, so a chat is answerable even in focus mode (§5) · a saved model that Ollama no longer has is flagged at connect time, not as a cryptic error mid-brew (§13).

2. Setup — one command, or the second-port method

C0fi needs two things reachable from the browser: your Ollama model server (with CORS allowed) and — for real web search / page reading — the local c0fi_server.py engine. There's now a launcher that starts everything and opens the app for you.

The fast path — start.sh

From the C0fi folder:

./start.sh                    # engine (search + read + MCP tools), opens the app
./start.sh --with-page-agent  # also starts the page-agent bridge (browser actions)

It launches c0fi_server.py on port 8790, opens the newest c0fi-v*.html in your browser, and stops every child process it started when you press Ctrl+C. If Ollama isn't answering it tells you the exact command to start it. That's the whole core stack in one terminal instead of one process per window.

One engine, one file, one portEarlier versions needed two separate Python programs (a big server.py plus an mcp_bridge.py that imported it) running in two terminals on two ports. v4.0 folded the only pieces C0fi actually used — real DuckDuckGo search and clean page reading — into a single zero-dependency c0fi_server.py that serves both the web endpoints and the MCP tools on port 8790. Nothing to install, nothing to import, one process.

Ollama with browser origins — the second-port method

C0fi runs in your browser, so Ollama must accept browser connections (CORS). On a machine that already runs the Ollama desktop app, don't fight whatever owns port 11434 — the app supervises its server and instantly respawns anything you kill. Sidestep it: run a second Ollama instance on its own port, just for browser tools.

  1. Leave the Ollama app (and anything else on 11434) exactly as it is.
  2. In a terminal, start a browser-facing instance on 11435:
OLLAMA_HOST=127.0.0.1:11435 OLLAMA_ORIGINS="*" ollama serve
  1. Open the C0fi HTML file → Engine settings → set the endpoint to http://localhost:11435 and press Enter (or click Test connection) — the dot goes green and the model list fills in without closing the dialog → pick a model → Save.
  2. The header dot goes green ("engine online"). Load a demo flow and hit ⏻ Brew.
Set it once v6.1Your endpoint, model, Verifier, temperature, and loop cap are saved in the browser and restored on every reload — connect once and it sticks. (A saved model your Ollama no longer has is caught at connect time with the list of what is installed, rather than failing with a raw 404 at Brew.)
Why this works every timeBoth instances share the same model store (~/.ollama), so every model you've ever pulled is available immediately — nothing re-downloads. The env vars are inherited straight from your shell, so no launchctl inheritance games, no kill/respawn race, no permission prompts. One tradeoff: keep that terminal window open — closing it stops the 11435 server. (start.sh checks both 11434 and 11435 and points you here if neither answers.)

Verify CORS is actually open (optional, from a second terminal tab):

curl -s -H "Origin: http://example.com" -D - http://localhost:11435/api/tags -o /dev/null | grep -i access-control
# want: Access-Control-Allow-Origin: *
Standalone mode — the file alone is capableWith nothing running but Ollama, C0fi can still research and write: Web Search climbs a ladder (local proxy → DuckDuckGo instant answers → a keyless jina.ai reader pass over DuckDuckGo's real results page) and Read Page falls back to the jina reader — each rung tried automatically, each fallback narrated in the Brew Log. c0fi_server.py, the stdio bridge, and Page Agent upgrade quality and add hands; they never gate basic capability. When a flow needs a tool you haven't set up, the Orchestrator completes everything achievable, hands you the artifact, and tells you exactly what to install for the rest.

Alternative: single Ollama instance on the default port

Only worth it on a clean machine or if you want zero extra terminals. The env var must reach the app, not just launchd — children inherit environment from their parent process at launch time.

launchctl setenv OLLAMA_ORIGINS "*"
killall Ollama        # capital O — quits the app, not just its server child
open -a Ollama        # fresh app inherits the setenv
Known trapIf you set launchctl setenv while Ollama.app was already running, respawned servers still carry the old environment — the app must be fully quit and relaunched. If killall Ollama and the port keeps getting re-occupied by a new PID, that's the app supervising its child: quit the app itself, or just use the second-port method above and move on. Neither setting survives a reboot; re-run it after rebooting.
Model adviceThe Orchestrator asks the model to emit strict JSON. Mid-size instruct models (7B–14B+) handle this well; tiny models may chat instead of build. Decision nodes run at temperature 0 regardless of your global setting.

4. Canvas controls

  • Add a node — press on a palette card and release over the canvas.
  • Move — drag a node by its title bar.
  • Configure — click the ⚙ gear on any node header, or double-click the title bar, to open the config drawer. Edits apply live.
  • Connect — click an out port (right side), then an in port (left side) on another node.
  • Cut a wire — click the wire.
  • Delete a node — select it and press Del, or use the drawer's Delete button. Its wires go with it.
  • Undo / redo v6.1 — ⌘/Ctrl+Z undoes, ⌘/Ctrl+Shift+Z (or Ctrl+Y) redoes; the toolbar ↶ ↷ buttons do the same. Every gesture is one step — a whole Orchestrator build, a demo load, or an import undoes as a single action, and a burst of drawer edits on one node coalesces. (Inside a text field, ⌘+Z stays native text-undo.)
  • Copy / paste a node v6.1 — select a node and ⌘/Ctrl+C, then ⌘/Ctrl+V drops a clone (same type, name, and config) at a small offset.
  • Fan-out / fan-in — one output can feed many inputs; many outputs can feed one input. Siblings sharing one output port run concurrently (see §8). A plain node reached by several wires runs once per wire; to instead merge parallel producers into one array, use the ⇉ Gather node (§5). Loops back to earlier nodes are allowed (see the safety valve in §8).
  • Pan — drag empty canvas. Zoom — trackpad pinch or mouse-wheel (toward the cursor), or the − 100% + controls bottom-left.
  • Fit / reset — ⤢ fit frames the whole flow in one click (great for a wide flow you can't see at once); ⌂ resets to 100%. Shortcuts: Shift+1 fit, Home reset.
  • Hide panels — the chevron handles on the canvas's left/right edges collapse the Ingredients and Orchestrator panels so the canvas grows. [ / ] toggle each; F is focus mode (both at once). If an 💬 Interaction node is waiting while the Orchestrator panel is hidden, a reply bar floats onto the canvas so a live chat is still visible and answerable v6.1 (§5).
  • Minimap — the overview bottom-right shows the whole flow and a rectangle for your current view; click or drag it to jump around. Double-click to dismiss it; the ▢ map button brings it back.

5. Node reference

Model slots & per-node overrides v6.2Any node that calls a model (C0fi Think, Decision, Critic) has a Model field. Once the engine dot is green it's a dropdown of the models your engine actually has — pick one directly, or pick a slot: Main, Verifier, Model C, or Model D (the four fields in Engine settings). A slot means "use whatever model I put in that Engine-settings field," so a flow stays portable — set three different models in Verifier / Model C / Model D and a three-judge flow runs three genuinely different models without hardcoding any names. Leave the field on engine default and the node uses the main model.

⏻ Trigger start

Every flow starts here. payload is the initial text handed downstream. Multiple triggers run one after another in a single brew.

☕ C0fi Think c0fi

Sends its prompt to the local model; output flows downstream and streams into the Brew Log live as it generates. Templates supported everywhere:

{{input}}         → whatever arrived on the in port
{{memory.key}}    → any key on the shared blackboard

⑂ Decision c0fi

Asks the model a question and fires exactly one outgoing branch. Verdicts run on the Verifier model if one is set in Engine settings (or this node's own model override) — a second model with different weights than the maker, so it doesn't share the maker's blind spots. Double-click the node: the drawer shows exactly which model rules here and why. Write the question as explicit pass/fail tests, not vibes. branches is a comma list (e.g. A,B or YES,NO,ESCALATE) — each label becomes its own out port, correctly positioned even for custom labels. Branch selection is robust: an exact one-word reply wins outright; a verbose reply is resolved by its last-mentioned label, so a small model that reasons before answering still routes to the verdict it actually settled on. Runs at temperature 0.

When to reach for Code Branch insteadA Decision node spends a model call and can, occasionally, misjudge. If the choice is mechanical — "does this look like a URL", "is this list empty", "is this valid JSON" — use the ⎇ Code Branch below: it's instant, free, and 100% deterministic.

⎇ Code Branch js · new

A deterministic fork — the same "pick one branch" job as Decision, but decided by JavaScript with no model call. Your code is the body of function(input, memory) and must return one of the branch labels. Instant, free, and never wrong.

// branches: A,B
return /^https?:\/\//.test(String(input)) ? 'A' : 'B';

Use it for loop conditions and any lexical/structural routing. The verified-briefing and bench demos use it as the reliable join/loop primitive that an LLM-routed gate couldn't guarantee.

▦ For Each loop · new

A native loop over a list — replaces the old hand-rolled "queue in memory + pop node + LLM gate" pattern. Two out ports: each fires once per item (wire it to the body you want run per item), and done fires once after all items, carrying memory[aggregateKey] as its output. Set items to a template that resolves to a JSON array (or newline-separated text); set aggregateKey to the memory key your per-item body appends to.

Sequential by designItems run one at a time, in order — not concurrently like a node's sibling wires elsewhere. Deliberate: the body is usually the same local model N times, which would just queue at the model anyway, and serial keeps the per-item log readable. Unlike the queue-and-gate pattern it replaces, For Each does not burn the loop-safety valve per item, so you can iterate a long list without raising Max loop iterations.

⌕ Web Search web

DuckDuckGo search, no API key. Two sources per node: proxy (default) calls the local c0fi_server.py engine's /api/search on port 8790 for real result lists; ddg-api hits DuckDuckGo's Instant Answer API directly from the browser — works with zero local services, but returns only knowledge-panel blurbs, so it's empty for most ordinary queries. Output is a numbered list of titles and URLs, ready for a Think node to pick from.

▣ Read Page web

Fetches a URL and returns clean readable text. proxy mode uses the engine's /api/read extractor; jina mode uses the free r.jina.ai reader (no local server needed). It extracts the first URL found in its input, so an upstream node that answers with a URL wires straight in. Output is capped at 12K characters to protect downstream context.

⚙ Tool Server mcp

The universal socket: calls any tool on any MCP server over HTTP. Three behaviors: leave tool empty and it outputs the server's tool catalog mid-brew; open the node's drawer and press ⌕ discover tools to browse and cache the catalog (cached catalogs are fed to the Orchestrator, so C0fi can build with tools it just learned about); or set a tool name and JSON args to call it. {{input}} and {{memory.key}} inside args are JSON-escaped automatically, so quotes and newlines in payloads can't break the call. The discover list shows each tool's parameter schema (* = required) and clicking a tool scaffolds correct arguments. The cleanup field names a tool to fire with {} if the call fails (e.g. stop_task), so failures never leave orphaned jobs. The built-in engine advertises five tools — web_search, read_page, ask_llm, and the knowledge-base pair kb_search / kb_list (§6) — at http://localhost:8790/mcp. See §6.

⟳ Watch Task mcp

Fire-and-monitor for long-running tools — the right shape for browser agents and any job that outlives a polite HTTP request. It starts startTool (default execute_task) without betting everything on that one reply, then polls statusTool on an interval, logging each status change; when the job truly finishes, the real result is captured, logged with a save button, and passed downstream. Every failure exit — deadline (maxMins), task error, or you pressing Stop — automatically fires stopTool first, so nothing is ever left running unattended.

⇄ HTTP Request io

Calls any URL — public APIs or other local services on your machine. {{input}} works in the URL and body. JSON responses are auto-parsed; anything else passes through as text.

Browser CORS appliesC0fi runs in your browser, so remote APIs must allow cross-origin requests. Local services you control (Flask apps, ERPNext, your own tools) can set their own CORS headers and become first-class nodes.

ƒ Transform js

Raw JavaScript over the payload. Your code is the body of function(input, memory) and must return the new payload. Full access to the blackboard object.

// example: extract JSON field and stamp it
const d = typeof input === 'string' ? JSON.parse(input) : input;
memory.lastSeen = new Date().toISOString();
return d.title + ' — ' + d.url;

▤ Memory state

Reads or writes the shared blackboard. Modes: write (overwrite key with input), append (add input as a new line), read (output becomes the key's value; input is ignored). See §9.

✎ Critic Loop c0fi

A review→revise cycle with a memory for its own law. Verdicts run on the Verifier model; rewrites run on the maker — adversarial by construction, and rewrites stream live so you watch each draft form. Each round: (0) a deterministic screen checks the draft against the constraint ledger — literal reuse of a banned phrase fails instantly, no verifier call spent; (1) the verifier judges against the goal AND every standing constraint, returning offending phrases verbatim; (2) each verified phrase enters the ledger (a memory key, default flags) as phrase :: reason; (3) the maker rewrites under orders to satisfy every constraint and critique point before finishing. Up to maxRounds (capped at 6).

Flags become law, and the law persistsThe ledger survives across brews and exports with the flow — Thursday's brew starts knowing everything Monday's flagged. Multiple Critic nodes can share one ledger key for house style, or keep separate ones per content type. Read, edit, or seed it by hand through an ordinary Memory node.
Template-aware barsThe goal interpolates {{memory.key}} like any prompt — so a bar can literally contain ground truth: "every claim must be supported by this page: {{memory.page}}". That is how the Sourced article writer demos do local hallucination-checking.

💬 Interaction live · new

Turns a brew into a live session. When the flow reaches this node it pauses, shows a message in the Brew Log, and waits for you to type a reply — a reply box appears at the bottom of the run pane. Your reply flows out the port; the brew resumes. On the first turn it shows its greeting; on later turns it shows whatever arrives on its in-port (usually the previous answer). Wire it in a loop — Interaction → (do something with the reply) → back to Interaction — and you have a multi-turn chatbot where the whole conversation is one brew. Type the endWord (default bye) or press Stop to end.

Conversations don't hit the safety valveEach turn is human-gated, so the loop counter resets when the flow passes through an Interaction node — a chat can run for as many turns as you like. The valve still catches a runaway automated loop (one that never waits for a human) within a single turn. Pair this node with kb_search (§6) to build a "chat with your docs" bot — see the ④ Interactive & Chat demos in §12.
Visible in focus mode v6.1A waiting Interaction node now pulses a "✍ your turn" badge on its card (at any zoom), and when the Orchestrator panel is collapsed a reply bar floats over the bottom of the canvas — type and press Enter there just like the panel's bar. So a full-canvas or focus-mode session still shows that the chat has started and lets you answer without reopening the panel. The full transcript stays in the Brew Log.

⇉ Gather join · new

A join / barrier: it waits until every wire coming into it has delivered, then fires once with all those inputs collected into a single array. Wire several parallel producers into one Gather to merge their results — no manual counter, no shared memory key. Its output is a JSON array, so a downstream Think node reading {{input}} sees all the pieces at once.

Gather vs. the default fan-inBy default a plain node reached by several wires runs once per incoming wire — which is exactly what the "Count + Code Branch" join in the bench and self-consistency demos relies on, and what For Each + a Memory append key does for aggregation. Gather is the no-boilerplate alternative when you just want "combine these N outputs into one." All three coexist; reach for whichever fits.
Only wire always-active producers inGather fires when the count of arrived inputs equals the number of wires into it. If one of those wires comes off a Decision/Code-Branch path that doesn't fire, the count never completes and Gather waits out the brew. Feed it producers that always run (parallel Think/search/tool nodes), not the two arms of an either/or branch.

◎ Output end

Terminal collector. Prints the labeled final payload to the Brew Log and shows it on the node card.

6. MCP & the Tool Server — everything connects

c0fi_server.py is a zero-dependency engine (plain python3, no pip installs) that gives C0fi its web hands and doubles as an MCP server. Run it directly or let start.sh do it:

python3 c0fi_server.py            # default port 8790
python3 c0fi_server.py 8795       # any port, if the default is taken

It serves the web endpoints the Search/Read nodes use (/api/search, /api/read) and an MCP surface at POST /mcp advertising five tools — web_search, read_page, ask_llm, and the knowledge-base pair kb_search, kb_list — that any MCP client can call: C0fi's Tool Server node, Claude Desktop, anything speaking the protocol. Sanity check: open http://127.0.0.1:8790/mcp in a browser and you get a JSON probe listing the tools.

Why real search needs a local helper at allA browser page can't directly fetch duckduckgo.com/html or an arbitrary article and read the response — those sites send no CORS headers for browser JavaScript. c0fi_server.py fetches server-side (no CORS between two programs on your own machine) and re-serves the result to the browser with the right headers. That's the one job the browser genuinely can't do alone — everything else in C0fi runs in the HTML file.

RAG over your own files — the knowledge/ folder new

Drop .txt / .md files into the knowledge/ folder next to c0fi_server.py, and kb_search retrieves the passages most relevant to a query — each tagged with its source filename — so a flow can answer grounded in your documents instead of the web. Call it from a Tool Server node: tool kb_search, args {"query":"{{input}}","k":3}. kb_list shows what's available.

Collections keep knowledge bases separateThe knowledge/ folder can hold subfolders, each a named collection — its own knowledge base. Pass a folder argument to search one: {"query":"{{input}}","k":3,"folder":"coffee"}. C0fi ships three: coffee/ (a sample shop KB), cofi-guide/ (C0fi's own documentation), and user-docs/ (empty, for your files). This is what lets one engine serve several unrelated bases without their answers bleeding into each other — the Help desk demo (§12) even routes each question to the right collection automatically.
Point it at your own filesDrop your .txt/.md into knowledge/user-docs/, delete the placeholder README.md, and Brew the Chat with YOUR docs demo — it's already wired to that folder, nothing to edit. (Or make a new subfolder and set the tool node's folder arg to match.) Ask before adding files and the bot just tells you where to put them.
Lexical now, semantic optionalRetrieval is lexical by default (keyword/BM25-style scoring with light stemming) — zero setup, works the moment you add files, and suits FAQ-style docs where the answer shares words with the question. For semantic matching (better when wording differs), set USE_EMBEDDINGS = True near the top of c0fi_server.py and pull an embedding model first: ollama pull nomic-embed-text. It falls back to lexical if the embed model isn't available.
Restart the engine after adding filesAfter editing c0fi_server.py or adding documents, kill any old instance before restarting (lsof -ti :8790 | xargs kill, then ./start.sh) — a stale copy keeps answering without your new files or tools.
Port busy? Move, don't fightIf the engine reports its port is in use, something else owns it (a stale python3 -m http.server is a classic — it answers with an HTML 404 page instead of the JSON probe). Pass a different port and update the node URLs to match, or just tell the Orchestrator to. To identify a squatter: lsof -i :8790. To kill a python script by name: pkill -f c0fi_server.py (process names are python3, so a bare pkill c0fi_server.py matches nothing).

Pattern to copy: any local service you own — a Flask router, ERPNext, a benchmark harness — becomes a C0fi ingredient with a bridge like this one: a tools list, a run_tool dispatcher, and the same JSON-RPC handler. One universal socket instead of a custom node per service.

The stdio bridge — the rest of the MCP ecosystem

Most MCP servers in the wild speak stdio, which a browser cannot. mcp_stdio_bridge.py is the generic adapter: it spawns any stdio MCP server as a child and exposes it at POST /mcp with CORS. One shim, whole ecosystem:

python3 mcp_stdio_bridge.py 8794 -- npx -y @page-agent/mcp
python3 mcp_stdio_bridge.py 8795 -- npx -y @modelcontextprotocol/server-filesystem /tmp

Environment variables pass through to the child. Long tool calls are held open up to 10 minutes. Like every C0fi bridge, a busy port makes it tell you the next port to try rather than fight.

Hands: Page Agent — driving your real browser

C0fi's web nodes are eyes; Page Agent (MIT) is hands — a Chrome extension GUI agent that reads the DOM as text and lets an LLM click, type, and navigate real, logged-in pages. It can't be absorbed into the Python engine (it needs the extension and a browser tab), so it stays a separate, optional add-on wrapped as an MCP tool. Its server exposes three tools: execute_task (blocking, natural-language task), get_status ({connected, busy}), stop_task. The chain: C0fi → stdio bridge (8794) → @page-agent/mcp → hub tab (WebSocket, port 38401) → extension → the LLM configured in the extension's panel.

Setup, in order:

  1. Install Page Agent Ext from the Chrome Web Store (≥ 1.5.11 for MCP) and enable it.
  2. In the extension's config panel, set the LLM: base URL http://localhost:11435/v1, your Ollama model, any non-empty key. Prove it by typing a trivial task in the side panel first — the panel is the brain that MCP tasks will use.
  3. Launch the bridge — the easy way is ./start.sh --with-page-agent, or bare by hand: python3 mcp_stdio_bridge.py 8794 -- npx -y @page-agent/mcp
  4. In Chrome, open http://localhost:38401 so the launcher opens the hub tab with its ?ws= parameter. Keep exactly one hub tab.
  5. In C0fi: Tool Server node → http://localhost:8794/mcp → discover → get_status should report connected. Then run tasks via a ⟳ Watch Task node with cleanup/stopTool = stop_task.
Do NOT set LLM_* environment variables on the bridgeTheir MCP server forwards env config with every task, which triggers a configure-before-execute step in the extension that disposes the running agent mid-start — the task instantly returns "Task failed. Task aborted." Launched bare, no config rides along, and tasks run on the proven panel configuration. (The --with-page-agent launcher passes only the LLM base URL/model the page-agent server needs to connect, not the per-task config that triggers the race.)
Writing tasks that succeedNumbered steps, explicit STOP lines, and forbidding side-quests keep a local model on the road: "1. Open linkedin.com/feed. 2. Click Start a post. 3. Paste the following text. 4. STOP — do NOT click Post.\n\n{{memory.draft}}" Stage big jobs as multiple small tasks (each gets a fresh agent run) rather than one monolith. Keep the human on the final button for consequential actions.
Consent lives in ChromeConnection approval prompts (and the Auto-approve toggle) appear in the extension, not in the browser running C0fi — keep Chrome visible during action brews. Denial stops the brew cold; that is the kill-switch working.

7. The C0fi Orchestrator — the build loop

The right-hand panel is a conversation with the engine itself. On every message, C0fi receives a live snapshot of your canvas (all nodes, configs, wires, memory keys), so it always knows what exists before acting. It replies with structured actions it applies immediately:

ActionEffect
add_nodeCreates and places a node with full config.
connectWires two nodes, including specific decision/branch labels.
set_configEdits any existing node — including ones you built by hand.
delete_nodeRemoves a node and its wires.
runStarts a brew (only when you ask it to run/test).

It knows the full node set, including Code Branch, For Each, and the ⇉ Gather join, and is told to prefer a deterministic Code Branch over a model-routed Decision whenever the choice is mechanical, to use For Each instead of hand-rolling a queue, and to reach for Gather when it needs to merge several parallel producers. Things that work well to say:

  • "Build a flow that drafts a LinkedIn post from a topic, gates it on quality, revises until it passes, and archives the winner. Then run it."
  • "Add a decision after the summarizer: if the input mentions pricing, route to a rebuttal writer, otherwise to the archive."
  • "For each URL in the search results, read the page and extract the key facts, then write one combined summary."
  • "What does this canvas do? Where would it break?" (questions are fine — it answers without building)
Orange action cardsWhen C0fi builds or edits, an orange card in the chat lists what it did (e.g. Trigger → Summarize → Publish gate). If the layout lands awkwardly, drag nodes around — position is cosmetic, wires are truth.
✨ Refine — feed a critique back in new in v5.7Next to Send is a Refine button. Paste in behavioral feedback — a bug you saw, a wrong result, or a reviewer's critique (even one from another model) — and hit Refine. Instead of building from scratch, the Orchestrator diagnoses which node caused the behavior and patches it with the minimal set of edits, preferring to rewrite existing nodes over adding new ones. It's the same builder LLM, framed for fixing rather than creating. One honest caveat: whether the local model gets the fix right from a critique depends on the model — for a precise, known fix, editing the node yourself is still the surest path; Refine shines for iteration and for feedback you'd rather not hand-translate into node edits.
The architect check — builds get validated new in v5.9After every build or Refine, a deterministic validator checks the result — pure graph facts, no second model: it adds a missing Trigger, appends a missing {{input}} to nodes that forgot it, and (the newest part) catches any node the builder added but left unwired. When it finds dangling nodes it asks the Orchestrator to connect just those; anything it still can't wire is highlighted red with a note telling you to drag a wire in — it never silently guesses a connection. This is the "let code check, let models interpret" rule applied to the build step: the judging is deterministic and free, the model is used only to fix what the check identifies.

The try: row covers the everyday moves — create a research flow, edit or audit the current canvas, drive your browser agent. The langchain patterns: row asks the Orchestrator to build a canonical pattern from scratch — router chain, smart (Code) branch, judge & gate, map-reduce, multi-query RAG. 8. How a brew executes

  1. Node states reset; every Trigger fires in turn with its payload.
  2. Data flows along wires. A running node glows copper; finished nodes go green; failures go red and stop that branch with the error in the Brew Log.
  3. Fan-out runs concurrently. When one output feeds several nodes, those siblings run in parallel — three judge models, say, all think at once instead of in sequence. Shared memory writes are safe (each is synchronous), but if two parallel siblings append to the same key, which lands first isn't guaranteed — use a Code Branch counter (the bench demo) or a ⇉ Gather node (§5) to join them when order matters.
  4. Decision and Code Branch nodes fire only the chosen branch's wires.
  5. For Each iterates its "each" body serially, then fires "done" once with the aggregated result (§5).
  6. Loop safety valve — cycles are allowed (e.g. output wired back to an earlier Think node). Each revisit counts as a loop pass; when passes reach Max loop iterations (Engine settings, default 8), that branch stops cleanly. The status bar shows loop pass n/max live. (For Each does not consume the valve per item; an Interaction node resets it each turn, so a chat can run indefinitely.)
  7. Interactive pause — an 💬 Interaction node stops the brew and waits for you to type; the brew resumes with your reply. A brew can therefore be a live, multi-turn session, not only a batch run (§5).
  8. Stop now aborts the in-flight model generation immediately v6.1 rather than only at the next node boundary — a long Think or Critic rewrite cancels the moment you press it. It also ends any Interaction node waiting for input.

9. The Memory blackboard

One shared key-value store lives for the whole session. It's the connective tissue that makes "everything can be connected" literal:

  • Memory nodes write / append / read it.
  • Any prompt, question, URL, or body can pull from it with {{memory.key}}.
  • Transform and Code Branch nodes get the live memory object and can read or mutate it directly.
  • It persists across brews within the session, and is included in flow exports — so a saved flow carries its accumulated state.

Pattern: run a research branch that appends findings to memory.notes, then a writer node whose prompt opens with "Context so far: {{memory.notes}}".

Two independent mechanisms — they don't overlap, and each does something the other can't:

The Insert option is the one Export/Import can't do at all: drop a saved cluster into whatever's already on the canvas and wire it in, so a saved flow works as a component, not just a snapshot. Rule of thumb — Export/Import is "save this to send or back up"; Save flow / My flows is "reuse this piece inside bigger flows I build later, without leaving the app."

Between sessions the browser keeps two things: your saved-flow library and (new in v6.1) your engine settings — the canvas itself is not persisted, so export before you close the tab if you want a portable copy of the current flow.

▶ Build App — ship a flow as a standalone app new in v5.3

The Build App button (header) turns the current flow into a standalone single-file app. Name it, and it downloads your-app.html — the same engine with the flow baked in and all the builder chrome (palette, canvas, Orchestrator, toolbar) hidden. Opening that file boots straight into a consumer UI, chosen automatically from the flow's shape:

  • Flow has a 💬 Interaction node → a chat app: a clean chat window that greets and waits for input, with engine logs hidden so only the conversation shows. Every RAG-chat flow becomes a shippable chatbot in one click.
  • Batch flow (Trigger → … → Output) → a form app: an input box seeded with the trigger's payload, a Run button, and a result panel showing the Output node's result.

The exported app has its own small header — app name, a live engine-status dot, a ⚙ to point at a different Ollama endpoint, and ↻ Restart. Under the hood it's this same page plus one injected line of config, so an app is even re-openable in the builder.

Lands with no engine? v6.2When someone opens your app on a machine with no model running, it now shows a short 3-line hint — start Ollama with OLLAMA_ORIGINS="*" ollama serve, then click ⚙ and point it at http://localhost:11434 — instead of just reading "engine offline." The hint clears itself the moment a model connects.
An app still needs a running engineA single HTML file can't bundle Ollama. The exported app needs Ollama reachable (and c0fi_server.py if the flow uses Search, Read, Tool, or Watch nodes) — the ⚙ lets whoever runs it set the endpoint. For a non-technical recipient, either hand over a folder with the app + c0fi_server.py + a start script, or point the flow's engine endpoint at a hosted LLM API before building so the file runs anywhere.
✨ Improve — the app can rebuild itself new in v5.7Because an exported app carries the whole engine (just hidden), it can revise its own flow. Every app has an ✨ Improve button in its header: describe what's wrong or what you'd change, click Revise & re-download, and the local model edits the embedded flow and downloads an updated version of the app — open the new file to run it. It's the same feedback loop as the builder's Refine, closed inside a shipped app. So the workflow becomes: build an app, watch it run, notice a flaw (or get a critique from another tool), paste that in, and get back a better app — no builder required. Same caveat applies: the quality of the revision depends on the local model.

11. Auto Brews — schedule flows to run automatically v6.8

Everything so far runs when you press ⏻ Brew. Auto Brews lets a flow run on its own, on a timer — turn "I click Brew every morning" into "it drafts my digest while I make coffee." Open it from the ⏰ Auto Brews button next to Clear canvas; the dialog is the list of every flow scheduled for automatic brewing.

How it runs — in the browser, tier 1Auto Brews are driven by this page while it's open — there's no server or daemon. They fire while the C0fi tab stays open and the machine is awake; closing the tab or sleeping the laptop pauses them. That's the honest limit of the in-browser approach (a headless "runs with the tab closed" runner is future work). Deployed brews persist across reloads, so reopening the file brings your list back — each keeps the paused state you set.

Deploying a flow

  1. Build the flow and press ⏻ Brew once to confirm it works.
  2. Open ⏰ Auto Brews → under Deploy the current canvas, give it a name, pick a schedule (below), and hit ＋ Deploy.
  3. It's now in the list and brews on schedule. The button shows ⏰ Auto Brews (active/total) and turns green when any are live.
ScheduleWhat it does
every N minutesA simple repeating timer — first run one interval after you deploy, then every N minutes. Good for "check this every 30 min."
daily at timeRuns once a day at a 24-hour HH:MM you set — e.g. 01:00 for 1 am or 14:00 for 2 pm.
weekly on days v6.8Runs at your HH:MM only on the days of the week you tick — any combination of Mon–Sun. "Every Monday at 1 am," "Mon/Wed/Fri at 09:00," "weekends at noon." Tick all seven and it reads every day. At least one day is required.

Picking daily at time or weekly on days reveals the Time (24h HH:MM) field; weekly also reveals the day-of-week checkboxes. It fires once within a two-minute window of the target time on each matching day, so a machine that's asleep or a tab that's closed at that moment simply misses that occurrence and catches the next one.

Deploying captures a full snapshot of the canvas (nodes + wires), so the scheduled flow is frozen at deploy time — you can keep editing the canvas without disturbing it. A flow must have a ⏻ Trigger to deploy (every brew starts from one). Deploy as many brews as you like; each runs independently on its own schedule.

Auto-saving results to a file

Just like the manual ⤓ save .txt on a node's output, an Auto Brew can save automatically. Tick ⤓ Auto-save each run's result to a .txt file at deploy time (or toggle it per row). Each run then drops a timestamped file — e.g. morning-digest-20260715-083000.txt — into your Downloads. Auto-save captures your ◎ Output node(s); a flow with no Output has nothing to save (add one to capture a result). Every run also stores its result in the list regardless, so ⤓ Save .txt is always available even with auto-save off. Because it's a browser, the file lands reliably only while the C0fi tab is focused.

How background runs stay out of your way

When a scheduled brew comes due, the engine snapshots your current canvas → swaps the scheduled flow in → brews it → restores your canvas. Whatever you had open comes back untouched, and a fresh Memory blackboard is used so a background run can't pollute your working state. It also won't hijack the canvas mid-edit — it skips a due run while you have a node drawer open or are dragging/wiring, and retries on the next tick.

One brew at a time — no collisionsA single engine lock means brews never overlap. If the engine is busy when one is due, C0fi holds and retries rather than colliding: a manual Run now retries in 2 minutes (up to five times); a scheduled brew simply waits for the next 15-second tick (it stays due, so it doesn't lose its turn). The canvas ⏻ Brew button is also refused while an Auto Brew holds the engine.
Keep scheduled flows fully automaticA flow with a 💬 Interaction node pauses waiting for a human — in a background run that just stalls the engine. Design Auto Brews to run end-to-end with no human turn: Trigger → work → Output.

12. Recipes & the demo library

The demo dropdown holds twenty-one flows in four groups. Each is a working, correct reference you can load, run, and take apart — and each of the LangChain-pattern additions rebuilds a canonical chain from C0fi's own nodes, so the pattern is legible instead of hidden inside a framework.

① Retrieval & RAG

  • Sourced article writer + v2 — Task → memory → query distillation → search → pick source → read page → grounded writer → Grounding editor → output. The writer may only use facts from the page it read; the Grounding editor is a Critic whose bar contains {{memory.page}} as ground truth. v2 adds two bar clauses (a verbatim-quote requirement with an escape hatch, and an explicit net-positive-reason requirement) to show exactly where a new quality rule gets added.
  • Verified multi-source briefing + v2 — gather several source URLs → for each: read + extract facts into a shared digest → cross-checked answer → Cross-source verifier (flags single-sourced claims, names disagreements). The original hand-rolls the loop (queue + pop + Code-Branch gate); v2 does the same with one For Each node. Load both and compare.
  • Multi-query retrieval LangChain MultiQueryRetriever — a Think node expands one question into three distinct search angles; a For Each searches each and collects every hit; a Merge node dedups and ranks. Pairs the "one question, many phrasings" idea with C0fi's For Each. Add a Read + grounded writer after the merge for full RAG.
  • Query decomposition LangChain decomposition — split a hard question into independent sub-questions; For Each researches and answers each on its own (a multi-node body per item); a Synthesize node combines them and names any tension.

② Chains, Routing & Agents

  • Content pipeline with a quality gate — Trigger → Think (draft) → Decision "publish? A/B" → A: Memory append → Output · B: Critic Loop → Output.
  • Router chain LangChain MultiPromptChain — a Decision node classifies the input (BILLING / TECH / GENERAL) and only the matching specialist Think node runs. The canonical "pick the right expert" chain. If your categories are keyword-detectable, a Code Branch classifier is instant and free.
  • Map-reduce summarize LangChain map_reduce — a Transform splits the document on blank lines; For Each summarizes each chunk (the map); a final Think fuses the chunk summaries into one (the reduce). This is how you summarize text longer than a context window — the model never sees the whole thing at once.
  • Agentic web browser — Question → search → pick best URL → read → coverage-gate Decision → answer from the page, with a fallback branch when the page is too thin.
  • MCP tool server tour — the same research shape, but every web action travels through the MCP engine's tools (web_search, read_page). The "everything connects" flow, end to end.

③ Quality, Judging & Loops

  • Self-improving loop — Think (attempt) → Critic → Decision "good enough? YES/NO" → NO wires back into Think with the critique; YES → Output. The safety valve caps runaway loops (§8).
  • Verifier bench + repair loop — one pitch fans out to three judges (Judge 1–3) with identical criteria but pointed at the Verifier / Model C / Model D slots — set three different models in Engine settings for genuinely independent judging — running in parallel; a Code Branch counts to three and opens the gate only once all verdicts are in (a deterministic join that survives concurrency); any RETRY routes to a Repair node that rewrites only the flagged phrases, then re-judges. Loops until unanimous or the safety valve.
  • Self-consistency vote self-consistency — a question is sampled three times in parallel; each sample writes a complete one-line ANSWER (the format instruction lives in the Sample prompt, so it survives whatever question you paste in), and the Extract node canonicalizes the whole answer: pure numbers normalize as numbers ($3,412.50 = 3412.5), and multi-part answers (e.g. a logic riddle) are lowercased, split, and sorted so order doesn't matter and a partial answer can't form a false consensus with a full one. A Code Branch waits for all three, then a tally majority-votes. Swap the Trigger for arithmetic, a riddle, or a classification. If you see 3/3 every time the task was too easy — raise Temperature to watch a slip get out-voted.
  • Self-consistency + verify new in v5.8 — the crucial follow-up. Self-consistency is a variance reducer, not a truth oracle: if the model is systematically wrong, the majority just launders the error into false confidence (a real run of a zoo-logic puzzle voted 2-of-3 for an answer that violated a rule). So after the tally, a deterministic rule-checker parses the winning answer and tests each puzzle rule in plain JS, and a Code Branch accepts only a valid answer — routing a wrong majority to a Reject node. It checks the answer against the rules, not the model. For a verifiable task, this is the pattern to steal: don't vote, verify. Rewrite the checker node with the checks your own problem needs.
  • LLM-as-judge eval LangChain evaluator — one model does a task; a separate model scores the result against a rubric and emits a SCORE line; a Code Branch gates on the number (≥4 accepts). Set a Verifier model so the judge has different weights than the maker. Wire the FAIL branch back for a score-gated retry loop.

④ Interactive & Chat v5.0–5.2

  • Interactive chatbot — Trigger (greeting) → 💬 Interaction → Think → back to Interaction. Brew it and the flow talks to you, waits for each reply, and answers — a whole multi-turn conversation as one live brew. Type bye or press Stop to end.
  • Chat with your docs — the flagship RAG chatbot: 💬 Interaction → remember the question → kb_search over the coffee collection → an LLM answers grounded only in the retrieved passages → back to Interaction. Ask "how late are you open Saturday?", then "can I return opened beans?", then "how much is membership?" — each answered from the sample files.
  • Chat with YOUR docs — the same shape, pre-wired to knowledge/user-docs/. Drop your own .txt/.md in, delete the placeholder README, restart the engine, and Brew — no node edits needed (§6).
  • Chat with the C0fi guide — dogfooding: kb_search over the cofi-guide collection (C0fi's own documentation). Ask "how do I connect Ollama?", "what does the Code Branch node do?", "how do I make a chatbot?" and get answers grounded in the guide.
  • Help desk — one bot, two knowledge bases, auto-routed: 💬 Interaction → a Decision classifies SHOP vs GUIDE → kb_search hits the matching collection → an LLM answers → back to you. Router chain + RAG + Interaction node together. Ask "how late are you open?" and it searches the shop KB; ask "what does For Each do?" and it searches the guide KB — same conversation.

Lessons the demos encode (learned the hard way)

Testable bars beat vibes. Close your lists: "exactly these six words and no others" beats "no hype words". Keep answer keys out of prompts. Let code count and match; let models interpret — a Code Branch counter never miscounts a join, an LLM gate can. And the discipline lives in the harness, not the weights: the same model that passes a judged loop reaches for "revolutionary" the moment it writes unjudged.

Local-stack bridge

Point HTTP nodes at services already on your machine — a Flask router, ERPNext endpoints, another Ollama model on a second port — and let Decision or Code Branch nodes route between them. C0fi becomes the conductor of your whole local rig.
